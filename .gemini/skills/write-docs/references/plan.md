# Planning Documentation Standards

이 문서는 설계(`DESIGN.md`), 계획(`PLAN.md`), 그리고 개별 작업(`TASK.md`) 문서의 작성 표준을 정의합니다. 실행 가능한 구체적인 가이드를 제공하는 것이 목표입니다.

## 1. 기능/리팩터링 설계서 (`DESIGN.md`)

특정 기능 구현이나 대규모 리팩터링을 시작하기 전, **기술적 접근 방식**을 정의합니다.

- **개요 (Overview)**: 변경의 목적과 범위.
- **아키텍처 변경 사항**: 모듈 구조의 변화나 의존성 수정을 시각화 (다이어그램 활용).
- **데이터 모델/구조**: 신규 구조체, 클래스 정의 또는 기존 모델의 변경 사항.
- **데이터 흐름 (Data Flow)**: 로직의 흐름이나 컴포넌트 간 상호작용의 변화.
- **기대 효과**: 설계가 가져올 성능 향상, 가독성 증대 등의 이점.

## 2. 구현 계획서 (`PLAN.md`)

설계된 내용을 바탕으로 **단계별(Phases) 및 작업별(Tasks)**로 분할한 로드맵입니다.

- **목표 (Goal)**: 이 계획을 통해 달성하고자 하는 최종 상태.
- **단계별 로드맵 (Phases)**: 논리적으로 구분된 단계별 목표.
- **작업 목록 (Task List)**: 각 단계에 속한 개별 작업들의 목록과 짧은 설명.
- **의존성**: 작업 간의 순서나 전제 조건 명시.

## 3. 작업 상세 문서 (`TASK-X-Y.md`)

하나의 원자적 작업을 수행하기 위한 **구체적인 실행 지침**입니다.

- **작업 목표**: 이 작업이 완료되었을 때의 구체적인 상태.
- **대상 파일**: 수정 또는 생성이 필요한 파일 경로 목록.
- **구현 상세 (Implementation Details)**:
    - 수정해야 할 코드의 위치와 구체적인 로직.
    - 복잡한 로직은 코드 블록을 활용하여 명확히 표현.
    - 기존 인터페이스와의 호환성 고려 사항.
- **검증 방법 (Verification)**:
    - 수행해야 할 테스트 케이스(유닛, 통합 등).
    - 기대되는 출력 결과나 동작 확인 방법.

---

## 4. 문서 작성 원칙 (Best Practices)

1.  **원자성 (Atomicity)**: 하나의 작업(Task)은 한 번의 실행 주기로 완료될 수 있는 크기로 쪼갭니다.
2.  **실행 가능성 (Actionability)**: 문서만 읽고도 추가적인 질문 없이 코드를 작성할 수 있을 정도로 구체적이어야 합니다.
3.  **검증 기반 (Test-Driven)**: 모든 작업에는 반드시 그 결과를 확인할 수 있는 테스트나 검증 단계가 포함되어야 합니다.
4.  **계층적 구조**: 설계 -> 계획 -> 작업 순으로 구체화하며, 상위 문서의 맥락이 하위 문서에 일관되게 전달되도록 합니다.
5.  **시각화**: 텍스트로 설명하기 복잡한 의존성이나 흐름은 다이어그램을 활용합니다.
