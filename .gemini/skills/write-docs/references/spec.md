# Documentation Standards (PRD/SPEC/DESIGN)

이 문서는 프로젝트의 각 문서가 담아야 할 핵심 내용과 작성 원칙을 정의합니다. 모든 문서는 Markdown 형식을 따르며, 상호 링크를 통해 정보의 연결성을 유지해야 합니다.

## 1. 전역 문서 (Global Documents)

### 1.1. `README.md` (Gateway)
프로젝트의 첫 인상이자 진입점입니다. 전문적인 기술 톤을 유지하며, 구체적인 **실행 워크플로우**, **구조 시각화**, 그리고 **실무적인 설정 예시**에 집중합니다.
- **프로젝트 소개**: 한 줄 요약 및 핵심 가치 (Bullet points).
- **사용 가이드**: 설치 방법 및 단계별 실행 워크플로우 (Code blocks).
- **프로젝트 구조**: 아스키 트리 등을 활용하여 소스 코드와 결과물의 배치를 시각적으로 표현.
- **설정 및 리소스 규격**: 주요 설정 파일 예시와 현실적이고 실무적인 리소스 정의 샘플 제공.
- **문서 가이드**: 상세 문서에 대한 링크를 `PRD` -> `DESIGN` -> `SPEC` 순서로 배치하여 가독성 확보.
- **라이선스**: 프로젝트 라이선스 명시.

### 1.2. `specs/PRD.md` (Requirements)
"우리는 무엇을 왜 만드는가?"에 대한 답을 제시합니다. 비기술적 관계자도 이해할 수 있는 언어로 작성합니다.
- **배경 및 목적**: 기존의 문제점과 이를 해결하려는 동기.
- **사용자 타겟**: 누가 이 제품/도구를 사용하는가?
- **주요 요구사항 (Functional Requirements)**: 시스템이 제공해야 할 핵심 기능 목록.
- **비기능적 요구사항**: 보안, 성능, 안정성 등에 대한 기대치.
- **성공 지표**: 프로젝트가 성공했다고 판단할 수 있는 정량적/정성적 기준.

### 1.3. `specs/SPEC.md` (Specifications)
"시스템이 외부와 어떻게 약속하는가?"를 정의합니다. 개발자가 기능을 구현하거나 사용자가 설정을 작성할 때 참조하는 **최종 규격**입니다. 이 문서의 최상위 제목은 `# Technical Specification`으로 작성합니다.
- **입력 규격/인터페이스**: 설정 파일의 스키마, API 정의, 데이터 포맷 등.
- **리소스/데이터 구조**: 폴더 구조, 파일 명명 규칙, 필수 요소.
- **처리 규칙**: 데이터 처리 알고리즘, 로직 우선순위, 병합 규칙 등.
- **제약 사항 및 유효성 검사**: 보안 규칙, 충돌 방지 정책 등 시스템이 강제하는 규칙.

### 1.4. `specs/DESIGN.md` (Internal Design)
"내부는 어떻게 구현되어 있는가?"를 설명합니다. 시스템의 유지보수와 확장을 위한 기술적 지도를 제공합니다.
- **시스템 아키텍처**: 전체적인 구조나 흐름을 시각화.
- **모듈별 역할 및 책임 (Separation of Concerns)**: 각 컴포넌트/모듈의 상세 역할. 모듈 목록에는 관련 문서(예: 모듈 README) 링크를 포함합니다.
- **데이터 모델 (Core Models)**: 핵심 구조체, 클래스, 데이터베이스 스키마 설계 의도.
- **의존성 그래프**: 모듈 간의 계층 구조 및 참조 방향.
- **핵심 알고리즘 설계**: 복잡한 로직의 내부 구현 방식에 대한 기술적 상세.

---

## 2. 모듈/컴포넌트 문서 (Module READMEs)

각 모듈 또는 패키지는 해당 영역의 **캡슐화된 정보**를 담는 `README.md`를 포함할 수 있습니다.

- **개요**: 모듈의 존재 이유와 시스템에서의 역할.
- **주요 구성 요소 (Components)**: 주요 클래스, 함수, 인터페이스 목록과 역할.
- **사용 예시**: 외부에서 이 모듈을 어떻게 사용하는지에 대한 코드 예시.
- **설계 의도**: 특정 설계 패턴이나 기술을 선택한 배경.

---

## 3. 작성 및 유지보수 원칙

1.  **단일 진실 공급원 (Single Source of Truth)**: 동일한 정보가 여러 문서에 중복되지 않도록 합니다. 규격은 `SPEC`에만 적고, 다른 문서에서는 링크를 사용합니다.
2.  **구현과 규격의 분리**: 사용자가 알아야 할 '동작'은 `SPEC`에, 개발자만 알면 되는 '구현 방식'은 `DESIGN`에 작성합니다.
3.  **최신화 유지**: 기능 변경 시 코드와 함께 문서를 즉시 업데이트합니다.
4.  **시각화 활용**: 아키텍처나 흐름도는 다이어그램을 활용하여 가독성을 높입니다.
